## Coding Workflow (Strict)

Non-trivial code = any task that touches more than one file, introduces new behavior, or involves CSS/JS/Python/SQL. (Single-file data-only changes like running `export_data.py` do not require grill-me.) Always goes  
through lazlo. Never use delegate_task for production code.

### Step 1: Write a task brief

Write the brief to `.claude/<descriptive-name>-task.md`. Include:
- Vault root path and all relevant file paths
- Design system file locations (tokens.css, trail.css, components.css)
- Exact list of files to modify vs files that are read-only
- Quality gates (grep checks, expected counts, etc.)
- The handback clause (see below)

**Always run the Prompt Engineer first for any lazlo brief.** The prompt-engineer  
subagent reads your raw task description and produces a structured, XML-tagged brief.  
It catches issues that vague prose misses. See the Prompt Engineer section below.

### Step 2: Run Grill-Me (before any code)

**EXCEPTION:** If a task brief already exists (e.g., `help-html-task.md`), treat it as capturing the grill-me outcome -- verify it still matches current project state (file paths exist, constraints unchanged, no new ADRs since written) before proceeding. Re-draft only if something has changed.

Before writing code, Vacation generates grill-me questions appropriate to the task -- as many as needed to fully think through the problem. Vacation answers all questions herself, drawing on project state, past ADRs, and her judgment of what Alex would want. She writes the full Q&A to `.claude/<task-name>-grillme.md` in Obsidian using this format:

- **Bold** questions Alex should at least glance at (UI/UX, user-visible behavior, real trade-offs)
- Plain text for internal technical decisions
- An "Alex's Thoughts:" section under every question (blank = approved, any text = discuss)

Vacation posts a brief note to Alex in Discord: "grill-me ready for review at `.claude/<task-name>-grillme.md`." Alex reviews in Obsidian and comments only where he disagrees. Silence = full approval.

**This is not a human-in-the-loop question session.** Vacation does the thinking. Alex is a reviewer, not an answerer.

### Step 3: Incorporate feedback, then launch implementation

After Alex reviews (or if he's silent, immediately), incorporate any overrides and launch lazlo:

```bash
export PATH='/usr/bin:/bin:/usr/sbin:/sbin:/opt/homebrew/bin:/Users/alex/.local/bin'
cd "/Users/alex/vaults/Vacation/Branson 2026"
claude --dangerously-skip-permissions \
  -p 'Read .claude/<task-name>-task.md. Grill-Me review is complete -- proceed to full implementation.' \
  --max-turns 100 --output-format json > /tmp/lazlo-output.json 2>/tmp/stderr.log
```

**PATH is mandatory.** Without it, claude returns "Not logged in" even though auth is fine.

### Step 4: Lazlo handback pattern

Every task brief must end with this handback clause:

```
When all code changes are complete, produce a structured handback report with these three sections, then STOP:

1. Files changed -- every file modified or created, one line each with a plain-English description.
2. Architectural choices -- any non-trivial decision (data flow, file ownership, pattern choice, constraint added/removed). Format each as: Choice / Why / Affects. If none, say so explicitly.
3. Invariants affected -- list any rule in CLAUDE.md, DECISIONS.md, or the task brief that was updated, relaxed, or newly discovered as a result of this session.

Do NOT commit, push, copy files, or update PROJECT_LOG. vacation-coordinator handles all post-code orchestration.
```

After lazlo finishes, YOU (vacation-coordinator) do: git commit, rsync to preview repo,  
path fix, cache-bust, push to Pages, update PROJECT_LOG.md, notify Alex.

### Step 5: Fresh-instance code review

Review must be a **fresh `claude -p` invocation** -- never `--continue` or `--resume` from  
the implementation session. The same instance that wrote the code cannot review it.

Write review brief to `.claude/<descriptive-name>-review.md`. Launch separately. Never chain.

### Iteration cap

3 rounds max (Hermes draft -> lazlo review = 1 round). If round 3 ends without  
sign-off, stop and surface to Alex with a written summary of every unresolved issue.

## GitHub Pages Deploy Workflow

Pre-push safety checks -- never skip:
```bash
grep -c 'pointerdown' web/quick-pick.html    # must return 1
grep -c 'fetch.*data.json' web/attractions.html  # must return >= 1
```

Full deploy sequence:
```bash
VAULT="/Users/alex/vaults/Vacation/Branson 2026"
PRODUCTION="/Users/alex/code/vacation-dashboard"

# 0a. Safety check -- NEVER skip
grep -c 'pointerdown' "$VAULT/web/quick-pick.html"    # must return 1 -- if 0: STOP
grep -c 'fetch.*data.json' "$VAULT/web/attractions.html"  # must return >= 1 -- if 0: STOP

# 0b. Regenerate data.json (REQUIRED -- idempotent, always safe to run)
python3 "$VAULT/scripts/export_data.py"

# 1. rsync (--exclude=".git" is MANDATORY -- omitting it silently destroys the repo)
rsync -av --delete \
  --exclude=".git" --exclude="DESIGN.md" --exclude="mockups" \
  "$VAULT/web/" "$PRODUCTION/"

# 2. Fix vault-relative paths -> GitHub Pages root paths (all 10 HTML files)
cd "$PRODUCTION"
sed -i '' 's|../assets/thumbs/|assets/thumbs/|g' \
  attractions.html shows.html index.html event-timeline.html people-timeline.html \
  wishlist.html suggested.html profile.html quick-pick.html help.html

# 3. Cache-bust CSS/JS asset URLs (run from $PRODUCTION -- script modifies files in cwd)
python3 "$VAULT/scripts/cache_bust.py"

# 4. Commit and push
git add -A && \
  git -c user.email="alexshultz@users.noreply.github.com" commit -m "MESSAGE" && \
  git -c credential.helper=osxkeychain push --force origin main
```

**If push fails with exit -1 and no output:** stale token in ~/.git-credentials.  
Always use `-c credential.helper=osxkeychain` explicitly.

**If .git is gone** (rsync --delete without --exclude=".git"):
```bash
cd "$PRODUCTION"
git init
git remote add origin https://github.com/alexshultz/vacation-dashboard.git
git -c credential.helper=osxkeychain fetch origin main
git config user.email "alexshultz@users.noreply.github.com"
git config user.name "Alex Shultz"
git checkout -b main
git add -A && git commit -m "restore" && git -c credential.helper=osxkeychain push --force origin main
```

## CSS Architecture (Never Violate These Rules)

- Components (`components.css`) reference semantic tokens only: `--color-bg`, `--accent-sand`  
  Never reference palette vars like `--moss` or `--lake` directly in components.
- Private palette lives in `css/themes/trail.css` only.
- To expose a palette color to components: add a passthrough token in `tokens.css`  
  (e.g. `--accent-sand: var(--sand)`) and use that in components.
- Any element toggled with `.hidden = true` that also has `display: flex` in components.css  
  MUST have a companion rule: `.element[hidden] { display: none; }`. Browser UA `[hidden]`  
  loses to author `display: flex` without this.

## Adding a New Page

1. Add `<link rel="preload" href="js/site.js" as="script">` to `<head>`
2. Add `<script src="js/site.js"></script>` as the FIRST child of `<body>`
3. Do NOT copy-paste static `<header>` or `<nav class="bottom-tabs">` HTML (site.js injects these)
4. If it is a sub-page (not in the main nav), add it to `NAV_ALIASES` in `site.js`
5. Add the new filename to the `sed` path-fix list in the deploy workflow Step 2
6. Add an ADR entry to `docs/DECISIONS.md`

## Markdown Line Breaks (Important)

Use two trailing spaces + newline to force line breaks within bullets or paragraphs.  
Standard single newline does NOT create a line break in Obsidian or GitHub Markdown.

```
This is line one.  
This is line two on a new line.
```

The two spaces at the end of line one are mandatory. This applies inside bullets too.
